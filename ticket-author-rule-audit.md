# Ticket Author — rule & workflow audit

Generated by the `ticket-author-rule-audit` workflow (63 agents; 13 of 14 rules judged — `full-urls` dropped, see caveats). Each rule was tested by a four-stage chain: design an adversarial probe → author a ticket under temptation following the skill → review it blind on sonnet (matching the shipped reviewer) → judge whether the rule was *followed*, *enforceable*, *redundant*, or *conflicting*. Five process/static agents tested the workflow itself.

## Headline

The rule set is largely sound: **9 of 13 tested rules are both followed under direct temptation and caught by the blind reviewer with a correct, exact-name citation.** The strongest layer is enforcement discipline — the reviewer cites rules by their bolded name, stays read-only, and is DRY against the skill via `skills:` injection. Weaknesses cluster in three places: (1) rule-vs-rule conflicts at "AC altitude", (2) the wiki/codebase anchor model, and (3) the self-validate/review loop.

## What works well

- **Use the language of existing documentation** — author kept `grace_period` under maximum temptation to coin "dunning runway"; reviewer caught the plant by name.
- **Reference files, not line numbers** — author chose content anchors over a permalink; reviewer flagged all three planted `#L` anchors by name. Grep-detectable.
- **Implementation Approach orients, not prescribes** — clean prose-vs-list discrimination; reviewer caught the planted numbered imperative list and left the authored prose alone.
- **Describe relationships in parts, not single verbs** — author decomposed every relationship; reviewer caught bare `Mirror…`/`Follow…`.
- **What the ticket carries depends on its anchor** — author resisted the strategic-priority temptation; reviewer cited the rule for the planted motivational paragraph. The strongest single discriminator in the set.
- **Acceptance Criteria assert outcomes** — all authored ACs falsifiable; reviewer caught all three planted sub-forms (restatement, unfalsifiable coverage, subjective boilerplate).
- **ODD tickets request resolution of an existing wiki ODD**, **File Naming** — both followed and caught by name.
- **Reviewer enforcement contract** — read-only, "you are not the author", template-driven required-vs-optional, cites by exact name, clean empty-batch sentinel.
- **CI as the authoritative cross-ticket gate** — `schema.py validate_epic_references` rejects >1 epic file and `epic:auto`-without-epic-file, with passing tests. The LLM layers are a soft pre-filter over a tested hard backstop.
- **Body-Rule → checkbox coverage is complete** — all 12 Body Rules have a backing Validation checkbox.

## Prioritized recommendations

### P0

1. **Resolve the AC-altitude conflict** between *Don't duplicate spec detail from the source doc* and *Acceptance Criteria assert outcomes*. The AC rule's "inline or via authoritative link" wording licenses the very duplication the sibling rule forbids — and this drove the only **actual authoring failure** in the test (`followed=false`). Add a tie-breaker cross-referenced from both rules: AC resolve spec-owned values via the authoritative link, not by reproducing the field/validation matrix inline; only a constant the AC's pass/fail genuinely turns on may be quoted. Ship a Good/Bad pair at field-table scale.

2. **Rewrite *Write actionable descriptions* into a falsifiable test.** This is the one rule that was available but **never fired** — the planted thin description ("X is wrong, see the wiki") went uncaught and only failed review on other grounds. The anchor and anti-duplication rules actively *reward* the thin-description pattern, and this is the sole counterweight. Replace "as you would a real issue" with a self-sufficiency standard: a reader recovers the concrete problem and done-state from Scope + AC alone, without opening the spec. Add the matching validation line and a note reconciling it with the anchor rules.

3. **Make the anchor model honest.** Identification keys on an unobservable counterfactual ("no wiki page would ever hold it"), the wiki/codebase split is presented as exhaustive when mixed tickets exist, and validation only checks internal consistency — so a consistent-but-wrong anchor passes. State the anchors are not mutually exclusive at the body level (a wiki-anchored ticket may carry mechanical sequencing); promote the observable motivational-rationale test to the primary identification rule; add a positive-direction validation checkbox that catches a codebase-labelled ticket that secretly needs a *why*.

4. **Fix the step-1 type-selection table.** Documentation is a first-class shipped type with no row — it falls through `No/No/No → Use Feature`, getting the wrong template and test discipline. The three boolean columns also encode an impossible cell (`ODD AND Spike/Bug/Chore`) and a self-contradicting ODD row. Reframe as a single mutually-exclusive decision flow that rules out Documentation and Epic before defaulting to Feature.

### P1

5. **Harden the self-validate/review loop (step 8).** It is unbounded with no escape hatch and no author/reviewer disagreement path, and its termination string `M of M tickets ready` (no period) never matches the reviewer's actual `N of M tickets ready.`. Add a re-review cap, surface unresolved violations to the user instead of looping, let the author decline a finding it can cite the wiki/code/template against, and key termination on the per-file `VERDICT: READY` lines.

6. **Align the reviewer's epic rule with CI.** The reviewer checks presence only ("an epic file is present"), so two epic files still pass and it returns READY on a batch CI rejects. Broaden to "at most one `type:epic` file; `auto` resolves to the single epic file", add the epic-linkage items missing from step-6 self-validation, and make "`epic` and `epic:` must not both be set on one file" a hard CI + reviewer rule (the unit tests already document this hole; `valid-epic.md` sets both and is asserted valid).

7. **Scope the snapshot-undecidable rules honestly.** Several rules read as auto-covered because `skills:` injection guarantees the reviewer *sees* them, but they can't be checked from one ticket without code/wiki access: terminology drift, *Do not infer technical decisions* (an invented `OrderRepository` reads identically to a real one), integer epic-IID existence, and spec duplication when `.claude/url-resolution.md` is absent — which it is in this repo today, so the duplication check is currently **disabled entirely** while a clean READY still implies "verified". Re-frame *Do not infer technical decisions* as "flag prescribed identifiers/paths lacking an anchor or user-request marker" (defer existence-verification), report the duplicate-spec check as "not performed" per file when a source is unresolvable, and ship a starter `url-resolution.md`.

### P2

8. **Polish the fuzzy-but-working boundaries.** Add behaviour-vs-case and bug-regression specificity guidance to *Testing names behaviours, not cases*; exempt anchored/spec-required behavioural detail in *Do not infer technical decisions* (it over-triggered on the authored ticket, colliding with actionability); fix the `weight` YAML-typing wording (bare unquoted integer ≥ 0; reject quoted digits, floats, `0`-as-unset); and reword the two orphan validation checks so each traces to a rule.

## Edge cases (boundaries the skill never draws sharply)

- **Anchor dichotomy** — mixed tickets (a migration with code-local sequencing *and* a domain *why*) have no home; the skill's own "Add email/phone to User" examples collide on exactly this.
- **Don't duplicate spec detail** — per-field AC at full schema scale reconstitutes the wiki table; the rule gives no heuristic for how many concrete values flips anchoring into copying.
- **Testing names behaviours** — "campaign expiry" (behaviour) vs "expiry at the end date" (case); the mandatory bug-regression clause collides with the no-enumeration clause.
- **Do not infer technical decisions** — committing to "an HTTP endpoint" vs GraphQL, or "a nullable read-time field" vs "a read boolean", sit on an unresolved line between capability and mechanism.
- **Frontmatter `weight`** — `"5"`, `3.0`, `0`, `-1` all sit at edges "non-negative integer" doesn't adjudicate at the scalar-type level.
- **ODD tickets** — re-scoping a broad wiki ODD into a narrow fork inside the ticket; the skill is silent on whether that counts as originating the decision.
- **Step-1 decision table** — `No / Yes / Yes` is internally contradictory, hidden only by a `—` wildcard.
- **Empty-batch handling** — step 9's "report ready once all READY" passes vacuously over zero verdicts; only step 8's special-case intercepts first.

## Gotchas (rule-vs-rule conflicts and misfires)

- **Don't duplicate spec detail vs Acceptance Criteria assert outcomes** — direct conflict; caused the one real authoring failure (see P0-1).
- **Write actionable descriptions vs What the ticket carries depends on its anchor** — the same Context line is "too much background" to one rule and "too little substance" to the other; the reviewer applied the anchor rule, masking the target violation.
- **Do not infer technical decisions over-triggers** — flagged anchored behavioural detail the Scope/AC already mandate, colliding with *Write actionable descriptions* and the user's implementation-ready demand. The author cannot be both implementation-ready and flag-free.
- **Testing vs Don't duplicate / Do not infer** — heavy overlap; an enumerated Testing section co-fires as a duplication finding, and the planted framework/path enumeration was routed to the wrong rule, shadowing the Testing rule's own no-frameworks clause.
- **Three-layer epic invariant drift** — CI (strongest) vs reviewer (weakest, presence-only) vs SKILL.md prose (no hook): the same rule stated at three strengths.
- **Termination string mismatch** — `M of M` literal vs `N of M tickets ready.` emitted.
- **Reviewer self-discovery** — `Glob proposed-tickets/*.md` picks up stale leftover files the current run didn't write; "review the tickets I just wrote" ≠ "everything in the folder".
- **url-resolution.md degrades silently** — when absent, the duplicate-spec check is skipped but READY still looks like a full pass.

## Ineffective rules (redundant or unenforceable)

- **Write actionable descriptions** — available but never fired (see P0-2).
- **Use the language of existing documentation (reviewer-side)** — structurally undetectable from one snapshot; deciding a term is "new" needs the whole vocabulary. Effectiveness was parasitic on the author having read the anchor wiki first.
- **Do not infer technical decisions (reviewer-side)** — can't confirm a name was lifted vs invented without code access; the violation is an *absence* of an anchor wrapped in normal prose — the highest-risk silent pass.
- **Step-1 ODD row** — self-contradicting: it only ever fires on an inferred ODD, exactly the case it declares illegitimate.
- **epic integer-IID validation (reviewer-side)** — half-enforceable; no tracker access, so a dangling IID passes silently.
- **Self-validate (step 6) duplicates the reviewer rubric** — same checklist applied twice by the same standard; the engine of oscillation on judgement-call rules.

## Coverage gaps

- **Orphan check** — Validation item "Risks entries point to concrete exposure" has **no backing Body Rule**; "Risks" appears nowhere else in SKILL.md. Either add a Risks Body Rule or remove the check.
- **Documentation mistyped as Feature** (see P0-4).
- **Anchor misidentification passes validation** — only internal consistency is checked, never correct choice; stripping the *why* to make the body consistent silences the very signal that would expose the error.
- **Self-validation has no epic-linkage item at all** — silent on the entire epic dimension.
- **Epic file carrying `epic:auto` self-satisfies** — `create.py` self-links via `epic=='auto'`, masking a missing real child; `valid-epic.md` documents the hole.
- **Don't-duplicate unenforceable without url-resolution.md** (absent here).
- **Reviewer strictly weaker than CI on multiple epics.**
- **Template-conditional sections gate on facts the reviewer can't read** — `User Story (only for user-facing features)` has no label signal; `Context (omit when part of an epic)` is decidable from frontmatter but falls outside "flag missing mandatory only".
- **NOTES has three meanings; step 8 handles one** — batch-pattern NOTES (the most actionable) and missing-file NOTES have no consumer.
- **Behaviour-changing chore needs no Testing** — chore Testing is optional and the reviewer flags missing mandatory only.

## Caveats on this audit

- **`full-urls` was not judged** — its judge agent finished without emitting structured output (one of 56 pipeline agents). Re-run just that rule: `Workflow` with `args: ["full-urls"]` on the saved script.
- **Reviews were run on inline ticket text**, not through the reviewer's real `proposed-tickets/` glob + `url-resolution.md` resolution. This tests the *rules*, not the file-I/O plumbing — so findings about URL-content duplication detection are weaker than a live run would show (and the report flags exactly that as a real gap).
- The harness reads the skill/reviewer/templates from disk each run, so after editing `SKILL.md` you can re-run the identical audit and diff the findings.
